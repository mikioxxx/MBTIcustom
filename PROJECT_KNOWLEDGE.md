# MBTIex Project Knowledge

このメモは、このプロジェクトで固まった設計・実装・運用の知見を、新しいMBTI系診断サイトへ流用するためのもの。

## 目的と体験設計

- 正確な心理検査ではなく、MBTIタイプを起点に「同じタイプの中でも自分は少し違う」と感じられるエンタメ診断。
- ユーザーが喜ぶ体験は「大勢に埋もれていない自分を再確認できる」こと。
- 診断名・称号・結果カードは、感傷的すぎず、でも表示されて嬉しい言葉にする。
- 「希少性」「個別性」「保存・共有したくなる見た目」を重視する。
- MBTIタイプ名は、普及している16Personalities系の通称をベースにした。
  - 例: INFP = 仲介者、ISFP = 冒険家。
- ただし公式ロゴや16Personalities名は使わない。

## 現在の公開構成

静的サイトとしてGitHub Pagesで動く構成。

```text
index.html
styles.css
app.js
16タイプ.png
data/
  app-config.json
  copy-templates.json
  mbti-profiles.json
  subtype-overrides.json
```

- `index.html`: 画面構造だけ。
- `styles.css`: UI、レスポンシブ、魔法陣風デザイン、結果カード表示。
- `app.js`: 診断ロジック、JSON読み込み、結果描画、共有処理、画像カード生成。
- `data/app-config.json`: 属性、MBTIタイプ一覧、設問。
- `data/mbti-profiles.json`: 16タイプ別の標準値、通称、属性ラベル、説明文。
- `data/subtype-overrides.json`: 個別調整済みの称号・説明。
- `data/copy-templates.json`: 汎用称号生成テンプレート、説明文パーツ。
- `ETC/`: 古い文書や未使用素材置き場。`.gitignore` で公開対象外。

## データ分離の方針

文言調整を人間がやりやすいように、診断文言・称号・タイプ設定はJSONへ分離した。

- HTMLに結果文や診断データを直書きしない。
- JSにはロジックだけを残す。
- 新しい診断パターンでは、まず `data/` のJSON設計を決める。
- 外部サーバーやGitHub Pagesに置くなら、JSON分離のままで問題ない。
- `file://` 直開きは `fetch()` がブロックされる可能性があるため、ローカル確認もHTTPサーバー前提。
  - 例: `python -m http.server 8000 --bind 127.0.0.1`

## 診断ロジック

現在の診断は、各設問の回答から6属性スコアを出し、選択MBTIタイプの標準値との差分を評価する方式。

属性:

```text
reality     現実
strategy    戦略
inquiry     探求
empathy     共感
imagination 想像
challenge   挑戦
```

流れ:

1. 各属性を初期値50で開始。
2. 回答値1-5を `answer - 3` に正規化。
3. 設問ごとの `weights` を使って属性へ加算。
4. スコアを `8-96` にクランプ。
5. 選択中のMBTIタイプの `expectedScores` と比較。
6. 差分が最大の属性を `primary` とする。
7. 差分が2番目に高い属性を `secondary` とする。
8. 差分が最小の属性を `contrast` とする。
9. `primary + contrast` の組み合わせで称号と説明を出す。

重要な判断:

- 生スコアの高さだけでなく、MBTIタイプごとの「予想値からの乖離」を見る。
- これにより、どのMBTIタイプでも同じ属性が勝ちやすい問題を避けた。
- `primary` だけでなく `contrast` も使うことで、称号に奥行きが出た。
- `primary x contrast` の組み合わせは多いが、事前生成・JSON管理なら運用可能。

## 結果生成

結果タイトルの基本形:

```text
本質を現実に結ぶ仲介者（INFP）
```

方針:

- 画面タイトルでは `通称（コード）` を出す。
  - 例: `仲介者（INFP）`
- 説明文では「理想主義者」などの通称より、`INFP` のようなコードを使うほうが診断感が出る。
- 「探求派のINFP」より「探求派の仲介者（INFP）」のように、ユーザーになじみやすい通称を使う。
- ただし前段の称号は「探求派」などの属性名をそのまま連結せず、意味ベースで自然な言葉にする。

結果で表示する情報:

- タイトル
- サブ説明
- 型バッジ
  - `強化型`: そのMBTIらしい高めの標準属性がさらに強い場合。
  - `変異型`: そのMBTIの標準像から少し外れた個性が出る場合。
- 主属性
- 副属性
- コントラスト
- 本文説明
- MBTI標準値との差分
- レーダーチャート
- 保存・共有用の診断結果カード

## UI/UXで固まった判断

### トップページ

- タイトルは `あなたのMBTIタイプ拡張診断`。
- `INFP subtype diagnostic` のような英字ピルは不要。スマホの縦スペースを浪費する。
- CTAはタイプ選択一覧より前に置く。
  - iPhoneではタイプ一覧の下に置くと、スクロールしないと開始できず操作性が悪い。
- MBTIタイプ一覧はスマホでも4列表示。
  - 2列だと縦長になりすぎる。
  - スマホではカードの余白・ラベルサイズを小さくして4列を維持する。
- 選択中タイプは明確にする。
  - 細い黄色枠だけでは弱い。
  - 未選択は少し暗く低彩度。
  - 選択中は明るく、太い金色リングを出す。

### 設問画面

- スマホでは選択後に自動で次へ進む。
- 選択肢は5段階。
- ラベルは左右の意味が分かる文言にする。
- 「正解はありません」系の説明は短く入れる。

### 結果ページ

- レーダーチャートは魔法陣風。
- 結果ページで属性が1つずつ点灯する演出を入れる。
- 差分バーは画面上には出すが、共有カードには入れない。
- 診断結果カードは3:4 PNGとして生成し、スマホで長押し保存できるよう `<img>` 表示する。
- `結果を共有` はWeb Share APIを使う。
  - 画像共有対応端末: PNG画像 + テキスト + URL。
  - テキスト共有のみ対応: 診断文 + URL。
  - 非対応: カード表示、画像を別タブで開く、共有文をクリップボードへコピー。

## 診断結果カード

目的:

- SNSで保存・共有したくなる「診断結果画像」。
- 長押し保存しやすい。
- サイトURLと一緒に共有できる。

仕様:

- Canvasで `1080 x 1440` の3:4 PNGを生成。
- 表示は `<img>` にData URLを入れる。
- 内容:
  - MBTIタイプ見出し
  - 結果タイトル
  - 型・主属性・副属性・コントラストのバッジ
  - 短いサブ説明
  - 本文説明
  - レーダーダイアグラム
- 入れないもの:
  - バーグラフ
  - 最下部の不要な英字タイトル

カード文字組みの注意:

- タイトルは折り返さず、1行に収まるようフォントサイズを自動調整。
- 本文は途中で `...` 省略しない。領域に収まるまで文字サイズと行間を調整。
- `INFP` など英数字は途中で `INF / P` のように割れないよう、英数字連続部分を1トークンとして扱う。

## デザイン方向

このサイトでうまくいった方向:

- 黒、チャコール、アンティークゴールド、 muted blue、ivory。
- 魔法陣、レーダーチャート、診断カード、奥行きのある暗い背景。
- SaaS的な無機質UIではなく、保存・共有したくなる占術/診断カード寄り。
- ただし可愛すぎるマスコット調や感傷的すぎるコピーは避ける。
- 装飾はリッチにするが、操作ボタンや設問は分かりやすく保つ。

避けるもの:

- ただのグラデーション背景。
- 説明的すぎる称号。
- 抽象的すぎる、感傷的すぎる称号。
- スマホでCTAが下に埋もれる構成。
- 画像カード内の長すぎる説明や細かすぎる文字。

## 共有と公開

公開URL:

```text
https://mikioxxx.github.io/mbitiex/
```

共有時は `publicSiteUrl` を定数として持ち、結果文と一緒に渡す。

GitHub PagesではHTTPSなので、Web Share APIや画像共有がローカルより安定する。

公開時の注意:

- 日本語ファイル名の画像でも動くが、将来的には英数字ファイル名の方が堅い。
- `ETC/` は公開しない。`.gitignore` に入れる。
- GitHub Pagesではリポジトリ名付きパスになるため、リソース参照は相対パスにする。
  - `fetch("data/app-config.json")`
  - `url("16タイプ.png")`

## 宣伝画像

作成済み:

```text
promo-poster.png
promo-poster-9x16.png
```

9:16版はSNSストーリー向け。黒金の魔法陣、中央の診断カード、CTAを大きく見せる方向。

ただし画像生成の日本語文字は崩れる可能性があるため、本番広告素材ではCanva/Figma/HTML Canvasなどで文字だけ後載せする案も検討する。

## 新プロジェクトへ展開する時の手順

1. まず新しい診断コンセプトを決める。
   - MBTIの何を拡張するのか。
   - ユーザーにどう嬉しく感じてもらうのか。
2. 属性軸を決める。
   - 6属性は流用可。
   - 新パターンなら属性名・意味を変えてもよい。
3. `expectedScores` の考え方を決める。
   - MBTIタイプごとの標準値からの乖離評価は流用推奨。
4. 設問を作る。
   - 回答値を属性に重み付けできるようにする。
5. 称号設計を作る。
   - `primary` と `contrast` の組み合わせで称号を作る。
   - 表示されて嬉しいが、意味不明・感傷過多にしない。
6. 画面UIを作る。
   - トップCTAはファーストビューに入れる。
   - スマホ優先でタイプ選択の縦長化を避ける。
7. 結果カードを作る。
   - 保存・共有前提。
   - バーグラフより、タイトル・説明・ダイアグラム・バッジ。
8. GitHub Pagesで公開。
   - 静的構成を守る。
   - 共有URLを定数化する。

## 実装で再利用しやすい関数

`app.js` で特に流用価値が高いもの:

- `loadAppData()`
  - JSONデータ分離の読み込み。
- `calculateScores()`
  - 設問回答から属性スコアを作る。
- `getDeviationScores(scores)`
  - MBTI標準値との差分評価。
- `renderTypePicker()`
  - 16タイプ選択UI生成。
- `drawRadar(scores)`
  - 魔法陣風レーダーチャート。
- `createResultCard(...)`
  - 3:4診断結果カードPNG生成。
- `drawFittedCanvasLine(...)`
  - カードタイトルを1行に収める。
- `drawFittedCanvasBlock(...)`
  - カード本文を省略せず枠内へ収める。
- `drawResultCardBadges(...)`
  - 型・属性・コントラストバッジ描画。

## 次回改善候補

- `app.js` が大きくなっているので、新規プロジェクトでは以下に分割するとよい。
  - `data-loader.js`
  - `scoring.js`
  - `result-card.js`
  - `charts.js`
  - `ui.js`
- 画像ファイル `16タイプ.png` は英数字名へ変更した方が公開運用では安全。
- 共有カードのデザインはHTML/CSSで一度プレビューし、それをCanvas化する方式にすると調整しやすい。
- iPhone実機の共有挙動はOS/ブラウザで差があるので、必要なら「画像を保存」「結果文をコピー」「サイトを共有」を分けた方が確実。
